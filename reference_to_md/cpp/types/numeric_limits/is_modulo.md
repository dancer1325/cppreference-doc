static const bool is_modulo; |  | (until C++11)  
---|---|---  
static constexpr bool is_modulo; |  |  (since C++11)  
| |   
  
The value of [std::numeric_limits](../numeric_limits.html)<T>::is_modulo is true for all arithmetic types `T` that handle overflows with modulo arithmetic, that is, if the result of addition, subtraction, multiplication, or division of this type would fall outside the range `[`[min()](min.html "cpp/types/numeric limits/min")`, `[max()](max.html "cpp/types/numeric limits/max")`]`, the value returned by such operation differs from the expected value by a multiple of max() - min() + 1. 

`is_modulo` is false for signed integer types, unless the implementation defines signed integer overflow to wrap.

### [[edit](https://en.cppreference.com/mwiki/index.php?title=cpp/types/numeric_limits/is_modulo&action=edit&section=2 "Edit section: Notes")] Notes

The standard said "On most machines, this is true for signed integers." before the resolution of [LWG issue 2422](https://cplusplus.github.io/LWG/issue2422). See [GCC PR 22200](http://gcc.gnu.org/bugzilla/show_bug.cgi?id=22200) for a related discussion. 

### [[edit](https://en.cppreference.com/mwiki/index.php?title=cpp/types/numeric_limits/is_modulo&action=edit&section=3 "Edit section: Example")] Example

Demonstrates the behavior of modulo types:

Run this code
    
    
    #include <iostream>
    #include <type_traits>
    #include <limits>
     
    template<class T>
    typename [std::enable_if](../enable_if.html)<[std::numeric_limits](../numeric_limits.html)<T>::is_modulo>::type
        check_overflow()
    {
        [std::cout](../../io/cout.html) << "max value is " << [std::numeric_limits](../numeric_limits.html)<T>::max() << '\n'
                  << "min value is " << [std::numeric_limits](../numeric_limits.html)<T>::min() << '\n'
                  << "max value + 1 is " << [std::numeric_limits](../numeric_limits.html)<T>::max()+1 << '\n';
    }
     
    int main()
    {
        check_overflow<int>();
        [std::cout](../../io/cout.html) << '\n';
        check_overflow<unsigned long>();
    //  check_overflow<float>(); // compile-time error, not a modulo type
    }

Possible output: 
    
    
    max value is 2147483647
    min value is -2147483648
    max value + 1 is -2147483648
     
    max value is 18446744073709551615
    min value is 0
    max value + 1 is 0

### [[edit](https://en.cppreference.com/mwiki/index.php?title=cpp/types/numeric_limits/is_modulo&action=edit&section=4 "Edit section: Defect reports")] Defect reports

The following behavior-changing defect reports were applied retroactively to previously published C++ standards. 

DR  | Applied to  | Behavior as published  | Correct behavior   
---|---|---|---  
[LWG 612](https://cplusplus.github.io/LWG/issue612) | C++98  | the definition of "handle overflows  
with modulo arithmetic" was poor[[1]](is_modulo.html#cite_note-1) | provided a  
better definition   
[LWG 2422](https://cplusplus.github.io/LWG/issue2422) | C++98  | `is_modulo` was required to be true for  
signed integer types on most machines  | required to be false for signed integer types  
unless signed integer overflow is defined to wrap   
  
  1. [↑](is_modulo.html#cite_ref-1) The definition is "adding two positive numbers can have a result that wraps around to a third number that is less". It has the following problems: 
     * It does not define the wrapped value. 
     * It does not state whether result is repeatable. 
     * It does not require that doing addition, subtraction and other operations on all values have defined behavior.
